# v1.3.8 Bugfix Investigation Notes

Session date: 2026-03-24/25

## Bug 1: OGG Vorbis files not recognized or playing (#74)

**Reporter:** GitHub issue #74
**Symptom:** OGG files placed in game music folders aren't recognized. No playback on either SDL2 or NAudio.

**Root Cause (two-part):**
1. `Constants.SupportedAudioExtensionsLowercase` was missing `.ogg` — files weren't discovered by `GameMusicFileService.GetAvailableSongs()`.
2. NAudio's `AudioFileReader` falls back to `MediaFoundationReader` for `.ogg`, which throws `COMException: 0xC00D36C4` ("byte stream type unsupported") on systems without Windows OGG codecs.

**Fix:**
- Restored `.ogg` in `Constants.SupportedAudioExtensions` and `SupportedAudioExtensionsLowercase`.
- Added `NAudio.Vorbis` (NVorbis 0.10.4) NuGet package for native OGG decoding.
- Created `Audio/OggFileReader.cs` — wraps `NVorbis.VorbisReader` as a `WaveStream` + `ISampleProvider`.
- NAudioMusicPlayer `Load()` and `PreLoad()` now use `OggFileReader` for `.ogg` files instead of `AudioFileReader`.
- SDL2 already had built-in OGG support via stb_vorbis (no changes needed).

**Files changed:** `Constants.cs`, `NAudioMusicPlayer.cs`, `Audio/OggFileReader.cs` (new), `UniPlaySong.csproj`

## Bug 2: Music never auto-plays — user must manually press play (Kadir's report)

**Reporter:** User Kadir (Discord)
**Symptom:** Switching to a game never starts music automatically. The top panel shows paused. User must manually click play every time. Affects ALL themes including default Playnite theme.

**Root Cause:** `ThemeOverlayActive` is a persisted setting (serialized to JSON) but is actually **transient runtime state**. Once set to `true` by any source (MusicControl Tag binding, Welcome Hub overlay, etc.), if Playnite exits before it's cleared back to `false`, the value persists on disk. On every subsequent launch, settings load with `ThemeOverlayActive=true`, the startup sync fires `HandleThemeOverlayChange(true)`, adds `PauseSource.ThemeOverlay`, and `ShouldPlayMusic()` returns `false` for every game selection. Nothing ever clears it because no `MusicControl` element exists in the default theme.

**Why it wasn't caught during development:** On setups with ANIKI REMAKE or similar UPS-integrated themes, the `MusicControl` element properly toggles `Tag=True/False` via DataTriggers, so the overlay clears promptly. The bug only manifests when the theme that *set* the flag is no longer active.

**Fix (two layers):**
1. **Startup reset (primary fix):** In `UniPlaySong.OnLoaded()`, reset `ThemeOverlayActive = false` and `VideoIsPlaying = false` before any playback logic runs. These are transient state that must never survive a restart.
2. **Runtime safety net (secondary):** Added 30-second stale overlay timeout in `ShouldPlayMusic()`. If `ThemeOverlayActive` has been `true` for >30s without being cleared, auto-clear it and log a warning. Tracks via `_themeOverlayActiveSince` timestamp set in `HandleThemeOverlayChange()`.

**Files changed:** `UniPlaySong.cs`, `MusicPlaybackCoordinator.cs`

## Bug 3: Open Log Folder fails on non-standard Playnite installs

**Reporter:** User Kadir (Discord)
**Symptom:** `DirectoryNotFoundException: "C:\Users\Kadir\AppData\Roaming\Playnite\Extensions"` — Kadir's Playnite is installed in `AppData\Local`, but the Open Log Folder command used `Environment.SpecialFolder.ApplicationData` (which resolves to `Roaming`).

**Fix:** Changed `OpenLogFolderCommand` to use `Assembly.GetExecutingAssembly().Location` to get the actual extension folder path, regardless of Playnite install type (standard vs portable vs custom path).

**Files changed:** `UniPlaySongSettingsViewModel.cs`

## Bug 4: Play/pause button shows wrong state (other user report, Helium theme)

**Reporter:** Another user (Discord)
**Symptom:** Song plays but top panel button shows stopped/paused. Happens randomly.

**Investigation:** Subagent analysis identified that `OnPlaybackStateChanged` fires in `AddPauseSource()` *before* the fader completes the actual pause (50-500ms later). The UI snapshots an intermediate state and never gets a follow-up notification when the fader finishes.

**Fix:** Added `OnFadeActionCompleted` callback on `MusicFader` that fires after the fader completes a pause/stop action. `MusicPlaybackService` hooks this to re-fire `OnPlaybackStateChanged`, giving the UI a second chance to read the final player state.

**Files changed:** `MusicFader.cs`, `MusicPlaybackService.cs`

**Status:** Fix applied but not confirmed by user. Low-risk (just an extra event fire). May not fully address the "random" nature of the bug — could also be theme-specific binding issues with Helium.
